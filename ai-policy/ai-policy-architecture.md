# Agentic AI Technical Policy
## Purpose
The following document defines the technical controls that limit the scope of AI agents in _Rolling_.
Generally, the AI layer for the API must remain bounded, non-privileged, analyst-initiated, and restricted to approved reporting tasks.

## Architectural Principle
* The intended flow for AI mode is:
    1. Core backend executes analysis normally
    2. The application builds a reduced AI-safe payload from approved outputs
    3. The AI service receives only that reduced payload
    4. The AI returns a more cohesive narrative report
    5. The analyst reviews the output.
* To support the distinction between traditional and AI reporting, there is a strict distinction between analysis outputs, namely:
    * structured analysis result payloads
    * traditional reports
    * AI-generated report drafts.

Output generated by AI is validated before storage or return. The validation checks:
* Output is text
* Output is non-empty if required
* Output has been recorded in the database
* Output is stored as draft content only
* Acceptance of output by an analyst is logged.

For AI request logging, only high-level workflow metadata is stored, along with the analysts who initiated and accepted/denied them. The logging contains:
* Route name
* Model name
* Request timestamp
* Project ID
* Analysis Run

## Approved AI Boundary
* AI agents may:
    * interpret structured descriptive analysis outputs
    * rewrite generated reports for clarity
    * summarise group findings by analyst request
    * propose follow-up analytical questions
* AI may not:
    * execute database queries directly
    * read arbitrary local files
    * invoke destructive scripts
    * choose or change permissions
    * operate without explicit analyst or route initiation.

### Enforced Boundary Module
AI functionality is centralised in a dedicated module: [`ai_reporting.py`](../app/services/ai_reporting.py). From this module, model calls can be made, prompts can be built, and AI outputs are processed.

There is no route, repository, or analysis engine module that calls the model directly. AI-facing code does not import or call repository functions directly.

The dedicated route for AI reporting is: `POST /analyses/descriptive-hormone/ai-report`. Otherwise, following analyst initiation, only internal services can call the AI reporting model.

## Approved Input Construction
* AI inputs are created through an explicit whitelist build:
    * `build_allowed_ai_input(...)`
    * `build_ai_report_prompt_input(...)`

The whitelisting functions will explicitly label fields, transforming them into models.

When unapproved fields are received, they will be rejected and logged.

* Prompt payloads will be constructed explicitly, like:
    * ```python
      prompt_payload = {
                        "summary": result.summary,
                        "tables": approved_tables,
                        "report_text": report_text,
                        "summary_text": summary_text,
                        "metadata": approved_metadata,
                        }
      ```

### Allowed AI Input Sources
* The AI input builders may use:
    * structured descriptive summaries
    * approved grouped tables
    * report texts
    * summary texts
    * minimal non-sensitive metadata needed for interpretation.

## Prohibited AI Input Sources
* The AI input builder must exclude
    * `.env` values
    * database credentials
    * secrets and authentication data
    * unrestricted file system contents
    * raw uploaded CSV files
    * record-level data not required for the reporting task

If the AI input builder cannot determine whether a field is safe, it will be excluded by default. Should an AI route receive insufficient approved data, an application error will be raised. The default policy is deny by default; only explicitly approved inputs are allowed.
